# GRACE Plan

Планирует фичу и формирует исчерпывающий Feature Specification Document.

## Что делает этот скилл

1. Анализирует запрос пользователя на фичу
2. Задаёт уточняющие вопросы через `question` tool
3. Формирует Feature Specification Document
4. Определяет компоненты, якоря и контракты
5. Связывает с `.kilocode/semantic-graph.xml`
6. Создаёт черновик плана тестов

## Параметры

```bash
/kilo skill grace-plan "описание фичи"
```

- `$ARGUMENTS` — описание фичи от пользователя

## Процесс планирования

### Шаг 1: Анализ запроса

Проанализируй:
- Какую проблему решает фича?
- Для кого она (какая роль пользователя)?
- Какой стек технологий используется?
- Есть ли зависимости от других фич?

### Шаг 2: Уточняющие вопросы

Используй `question` tool для уточнения:

**Обязательные вопросы**:
1. Какая бизнес-цель этой фичи?
2. Кто пользователи фичи (роли)?
3. Какие данные нужны на входе?
4. Какие результаты ожидаются?
5. Есть ли критичные ограничения (безопасность, производительность)?

**Контекстные вопросы**:
- Требуется ли интеграция с внешними системами?
- Есть ли специфичные требования к UI/UX?
- Какие edge cases нужно учесть?

### Шаг 3: Генерация Feature ID

Формат: `FEAT-{NUMBER}`

Примеры:
- FEAT-001
- FEAT-002
- FEAT-003

Проверь `plans/features/` для определения следующего номера.

### Шаг 4: Определение компонентов

Для каждого компонента фичи:

1. **Тип**: model, view, service, serializer, API endpoint, etc.
2. **Путь**: где будет расположен файл
3. **Зависимости**: от каких других компонентов зависит

Обнови `.kilocode/semantic-graph.xml`:

```xml
<component id="backend.apps.{module}.{component}" kind="{kind}" path="{path}">
  <role>{описание роли}</role>
  <depends-on ref="{DEPENDENCY_ID}"/>
  <exposes>
    <api name="{API_NAME}" type="{type}">{description}</api>
  </exposes>
</component>
```

### Шаг 5: Определение ANCHOR и контрактов

Для каждой критичной функции:

1. **ANCHOR_ID**: `UPPER_SNAKE_CASE` идентификатор
2. **Функция**: имя функции в коде
3. **Файл**: путь к файлу

Контракт должен содержать:

| Поле | Описание |
|------|----------|
| PURPOSE | Одна фраза — зачем функция |
| @PreConditions | Условия, которые ДОЛЖНЫ быть истинны |
| @PostConditions | Гарантии результатов |
| @Invariants | То, что НЕ ДОЛЖНО нарушаться |
| @SideEffects | Побочные эффекты или "нет" |
| @ForbiddenChanges | Что нельзя менять |

Пример:

```python
# [START_AUTH_LOGIN]
"""
ANCHOR: AUTH_LOGIN
PURPOSE: Аутентификация пользователя по email и паролю.

@PreConditions:
- email: валидный email формат
- password: непустая строка

@PostConditions:
- при успехе: { access_token, refresh_token, user }
- при ошибке: { error: "INVALID_CREDENTIALS" }

@Invariants:
- пароль никогда не возвращается в ответе

@SideEffects:
- создаёт/обновляет refresh_token в БД
- пишет лог попытки входа

@ForbiddenChanges:
- нельзя убрать rate limiting
- нельзя возвращать разные ошибки для несуществующего пользователя и неверного пароля
"""
```

### Шаг 6: Формирование Feature Spec

Создай файл `plans/features/FEAT-XXX.md` по шаблону `.kilo/templates/feature-spec-template.md`.

Обязательно заполни:

- [ ] Описание и цель
- [ ] Новые и изменяемые компоненты
- [ ] Зависимости и связи
- [ ] Контракты для всех ANCHOR
- [ ] User Stories с Acceptance Criteria
- [ ] Черновик тест-плана

### Шаг 7: Обновление статуса

Установи статус `planning` в Feature Spec.

## Пример использования

```bash
/kilo skill grace-plan "Логин пользователя по email и паролю с JWT аутентификацией"

/kilo skill grace-plan "CRUD операции для курсов: создание, редактирование, удаление, публикация"

/kilo skill grace-plan "Система уведомлений о новых курсах для пользователей"
```

## Результат

После выполнения:

- [ ] Файл `plans/features/FEAT-XXX.md` создан
- [ ] Все ANCHOR определены
- [ ] Контракты для всех ANCHOR написаны
- [ ] `.kilocode/semantic-graph.xml` обновлён
- [ ] User Stories определены
- [ ] Статус: `planning`

## Что делать после

1. Проверь Feature Spec на полноту
2. Запусти `grace-validate-plan FEAT-XXX` для валидации
3. При необходимости — уточни Feature Spec

## Чеклист качества Feature Spec

- [ ] Цель фичи понятна и измерима
- [ ] Все компоненты имеют пути в проекте
- [ ] Для каждой функции есть ANCHOR и контракт
- [ ] Контракты содержат все обязательные поля
- [ ] User Stories имеют измеримые критерии
- [ ] Зависимости от других фич указаны
- [ ] Semantic graph обновлён
