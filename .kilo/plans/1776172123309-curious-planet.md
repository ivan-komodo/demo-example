# План: Создание системы скиллов GRACE для Kilo

> **Версия**: 1.0
> **Дата**: 2026-04-14
> **Цель**: Создать 6 скиллов для методологии контрактов, логирования и семантической разметки

---

## 📋 Обзор

Создаём систему скиллов **GRACE** (Generic Rule-based Architecture for Contracted Execution) для работы по методологии проекта:
- Semantic code markup с ANCHOR и контрактами
- AI-friendly логирование
- Валидация через тесты 3 уровней
- Итеративная разработка фич

## 🤖 Агенты GRACE

### implementer.md

**Режим**: primary
**Модель**: текущая code-agent модель

**Назначение**: Реализация фич по спецификации

**Ответственность**:
- Чтение Feature Spec и Validation Report
- Генерация кода с семантической разметкой
- Создание unit-тестов
- Обновление semantic-graph.xml

**Права**:
```json
{
  "edit": {
    "backend/**": "allow",
    "frontend/**": "allow",
    "tests/**": "allow",
    ".kilocode/semantic-graph.xml": "allow",
    "plans/features/*.md": "allow"
  },
  "read": "allow",
  "bash": {
    "pytest": "allow",
    "npm test": "allow",
    "python manage.py": "allow"
  }
}
```

---

### contract-reviewer.md

**Режим**: subagent
**Назначение**: Проверка корректности контрактов в коде

**Ответственность**:
- Парсинг ANCHOR комментариев в коде
- Проверка обязательных полей контракта
- Сравнение контрактов в коде с контрактами в Feature Spec
- Выявление расхождений

**Вывод**: Отчёт о соответствии контрактов

---

### verification-reviewer.md

**Режим**: subagent
**Назначение**: Проверка документов и тест-планов

**Ответственность**:
- Проверка полноты Feature Spec
- Валидация Validation Report
- Проверка тестовых сценариев
- Формирование Verification Report

**Вывод**: Pass/Fail с рекомендациями

---

### fixer.md

**Режим**: primary
**Назначение**: Исправление багов

**Ответственность**:
- Анализ Verification Report
- Локализация проблемы через лог-маркеры
- Проверка нарушений контракта
- Исправление кода
- Обновление тестов

**Права**:
```json
{
  "edit": {
    "**": "allow"
  },
  "read": "allow",
  "bash": "allow"
}
```

## 📦 Детализация по скиллам

### 1. grace-init

**Назначение**: Инициализация проекта по методологии GRACE

**Что делает**:
1. Создаёт структуру директорий проекта
2. Генерирует `.kilocode/semantic-graph.xml` (если не существует)
3. Создаёт `.kilocode/rules/` с базовыми правилами
4. Инициализирует примеры семантической разметки
5. Создаёт `.kilo/` структуру для skills/agents/commands
6. Генерирует базовый `AGENTS.md` или `.kilo/instructions.md`
7. Создаёт `plans/` директорию для ТЗ и планов

**Артефакты**:
- `.kilocode/semantic-graph.xml` — семантический граф
- `.kilocode/rules/` — правила методологии
- `AGENTS.md` — инструкции для агентов
- `.kilo/skill/` — директория для скиллов

**Параметры**:
- `--project-name` — имя проекта
- `--stack` — стек технологий (django, node, go, etc.)
- `--minimal` — минимальная инициализация (только правила)

---

### 2. grace-plan

**Назначение**: Планирование фичи с формированием ТЗ

**Что делает**:
1. Анализирует запрос пользователя на фичу
2. Задаёт уточняющие вопросы через `question` tool
3. Формирует Feature Specification Document
4. Определяет компоненты, якоря и контракты
5. Связывает с semantic-graph.xml
6. Создаёт черновик плана тестов

**Артефакты**:
- `plans/features/{feature-id}.md` — ТЗ фичи (по шаблону)
- Обновление `.kilocode/semantic-graph.xml` (новые компоненты)

**Шаблон Feature Spec**:
```markdown
# Feature: {название}

## 📋 Описание
{краткое описание фичи}

## 🎯 Цель
{зачем нужна эта фича}

## 📦 Компоненты
{список модулей/файлов/якорей}

## 🔗 Связи
{зависимости от других компонентов}

## 📝 Контракты
{список ANCHOR-ID с краткими description}

## ✅ Критерии приёма
{user stories и acceptance criteria}

## 🧪 Тест-план
{ссылка на validation report}
```

**Параметры**:
- `$ARGUMENTS` — описание фичи от пользователя

### 3. grace-validate-plan

**Назначение**: Валидация полноты и непротиворечивости ТЗ

**Что делает**:
1. Проверяет наличие всех обязательных секций в Feature Spec
2. Валидирует контракты на полноту (ANCHOR, PURPOSE, Pre/Post/Invariants)
3. Проверяет связи в semantic-graph.xml
4. Формирует Validation Report с тест-планом 3 уровней:
   - **Детерминированные тесты** — проверки результатов
   - **Тесты траектории** — проверки log-маркеров
   - **Интеграционные тесты** — E2E сценарии

**Артефакты**:
- `plans/validation/V-{feature-id}.xml` — отчёт валидации

**Формат Validation Report**:
```xml
<?xml version="1.0" encoding="UTF-8"?>
<validation-report feature-id="{id}" date="{date}">
  <status>{pass|fail|incomplete}</status>
  
  <completeness>
    <section name="description" status="pass"/>
    <section name="contracts" status="pass"/>
    <section name="acceptance" status="fail" reason="missing criteria"/>
  </completeness>
  
  <contracts>
    <contract anchor="{ANCHOR_ID}">
      <has-purpose>true</has-purpose>
      <has-preconditions>true</has-preconditions>
      <has-postconditions>true</has-postconditions>
      <has-invariants>true</has-invariants>
    </contract>
  </contracts>
  
  <test-scenarios>
    <scenario id="SC-001" level="deterministic">
      <description>Валидный логин возвращает сессию</description>
      <given>Пользователь существует</given>
      <when>POST /auth/login с валидными credentials</when>
      <then>expect(result.status).toEqual(200)</then>
    </scenario>
    
    <scenario id="SC-002" level="trajectory">
      <description>Проверка лог-маркеров логина</description>
      <required-markers>
        <marker point="ENTRY">[Auth][login][BLOCK_VALIDATE_INPUT]</marker>
        <marker point="CHECK">[Auth][login][CHECK_USER_EXISTS]</marker>
        <marker point="EXIT">[Auth][login][EXIT_SUCCESS]</marker>
      </required-markers>
    </scenario>
    
    <scenario id="SC-003" level="integration">
      <description>E2E сценарий авторизации</description>
      <steps>
        <step order="1">Пользователь открывает страницу логина</step>
        <step order="2">Вводит credentials</step>
        <step order="3">Получает JWT token</step>
        <step order="4">Может обратиться к protected endpoint</step>
      </steps>
    </scenario>
  </test-scenarios>
</validation-report>
```

**Параметры**:
- `$1` — путь к Feature Spec или ID фичи

### 4. grace-execute

**Назначение**: Генерация кода по ТЗ фичи

**Что делает**:
1. Читает Feature Spec и Validation Report
2. Генерирует код с семантической разметкой:
   - ANCHOR и CHUNK комментарии
   - Полные контракты для всех функций
   - AI-friendly логирование (ENTRY, EXIT, BRANCH, DECISION, ERROR)
3. Создаёт unit-тесты по тест-сценариям
4. Обновляет статус фичи в Feature Spec
5. Обновляет semantic-graph.xml

**Артефакты**:
- Исходный код с разметкой в соответствующих директориях
- Тесты в `tests/` директории
- Обновлённый `plans/features/{feature-id}.md` (статус: implemented)

**Требования к коду**:
- Каждый критичный блок имеет `[START_ANCHOR]` ... `[END_ANCHOR]`
- Контракт содержит все обязательные поля
- Логирование использует `logLine()` с тем же ANCHOR_ID
- Нет декоративных логов без диагностической ценности

**Параметры**:
- `$1` — путь к Feature Spec или ID фичи

---

### 5. grace-verification

**Назначение**: Прогон всех тестов и проверка работоспособности

**Что делает**:
1. Запускает все тесты (unit, integration, e2e)
2. Парсит логи на наличие обязательных маркеров
3. Проверяет покрытие тестами
4. Формирует отчёт верификации
5. При падении — помечает фичу как `failed-verification`

**Артефакты**:
- `reports/verification-{timestamp}.md` — отчёт верификации
- Обновлённый `plans/features/{feature-id}.md` (статус: verified / failed)

**Формат Verification Report**:
```markdown
# Verification Report

**Feature**: {id}
**Date**: {timestamp}
**Status**: {passed|failed}

## Test Results

### Unit Tests
- Total: 15
- Passed: 14
- Failed: 1

### Integration Tests  
- Total: 5
- Passed: 5
- Failed: 0

## Log Markers Check

| Anchor | Expected Markers | Found | Status |
|--------|------------------|-------|--------|
| AUTH_LOGIN | ENTRY, EXIT, CHECK | All | ✅ |
| AUTH_REGISTER | ENTRY, EXIT, DECISION | Missing DECISION | ❌ |

## Coverage
- Statements: 82%
- Branches: 78%
- Functions: 85%

## Issues
1. Missing log marker in AUTH_REGISTER
2. Test auth.test.js:45 fails with unexpected error
```

**Параметры**:
- `$1` — путь к Feature Spec или ID фичи (опционально, если нет — все фичи со статусом `implemented`)

---

### 6. grace-fix

**Назначение**: Исправление багов с обновлением тестов

**Что делает**:
1. Анализирует отчёт верификации или принимает описание бага
2. Локализует проблему через лог-маркеры
3. Проверяет контракт функции на нарушения
4. Исправляет код, сохраняя семантику контракта
5. Обновляет тестовые сценарии
6. Добавляет новые тесты для регрессии
7. Обновляет Validation Report

**Артефакты**:
- Исправленный код
- Обновлённые тесты
- `reports/fix-{bug-id}.md` — отчёт исправления
- Обновлённый `plans/validation/V-{feature-id}.xml`

**Формат Fix Report**:
```markdown
# Fix Report

**Bug ID**: {id}
**Feature**: {feature-id}
**Date**: {timestamp}

## Problem
{описание проблемы}

## Root Cause
{анализ причины через лог-маркеры}

## Contract Violation
{какой контракт был нарушен}

## Fix
{описание исправления}

## Tests Added
1. test_bug_{id}_regression.py — проверяет что баг не повторится
2. test_contract_anchor.py — проверяет инвариант контракта

## Updated Markers
- Added: [Auth][login][CHECK_SUSPICIOUS_ACTIVITY]
- Modified: [Auth][login][DECISION] now logs reason
```

**Параметры**:
- `$1` — ID бага или путь к Verification Report
- `$ARGUMENTS` — описание бага (если нет отчёта)

```xml

<validation-report 
  feature-id="{FEATURE_ID}"
  feature-name="{FEATURE_NAME}"
  date="{DATE}"
  validator="{AGENT_ID}">
  
  <!-- === СТАТУС ВАЛИДАЦИИ === -->

  <summary>{краткое резюме}</summary>
  
  <!-- === ПРОВЕРКА ПОЛНОТЫ === -->
  <completeness>
    <section name="description" status="{pass|fail}">
      <check item="present" status="{pass|fail}"/>
      <check item="clear" status="{pass|fail}"/>
    </section>
    
    <section name="components" status="{pass|fail}">
      <check item="paths_defined" status="{pass|fail}"/>
      <check item="types_defined" status="{pass|fail}"/>
    </section>
    
    <section name="contracts" status="{pass|fail}">
      <check item="all_anchors_defined" status="{pass|fail}"/>
      <check item="all_fields_present" status="{pass|fail}"/>
      <issue anchor="{ANCHOR_ID}">
        {описание проблемы}
      </issue>
    </section>
    
    <section name="acceptance" status="{pass|fail}">
      <check item="user_stories_defined" status="{pass|fail}"/>
      <check item="criteria_measurable" status="{pass|fail}"/>
    </section>
  </completeness>
  
  <!-- === ПРОВЕРКА КОНТРАКТОВ === -->
  <contracts>
    <contract anchor="{ANCHOR_ID_1}">
      <function>{function_name}</function>
      <file>{path}</file>
      <has-purpose>true</has-purpose>
      <has-preconditions>true</has-preconditions>
      <has-postconditions>true</has-postconditions>
      <has-invariants>true</has-invariants>
      <has-sideeffects>true</has-sideeffects>
      <has-forbidden-changes>true</has-forbidden-changes>
      <issues>
        <!-- Проблемы если есть -->
      </issues>
    </contract>
    
    <contract anchor="{ANCHOR_ID_2}">
      <!-- ... -->
    </contract>
  </contracts>
  
  <!-- === ПРОВЕРКА КОНСИСТЕНТНОСТИ === -->
  <consistency>
    <check name="semantic_graph_refs" status="{pass|fail}">
      {все ссылки в контрактах есть в semantic-graph.xml}
    </check>
    <check name="no_duplicate_anchors" status="{pass|fail}"/>
    <check name="dependencies_valid" status="{pass|fail}"/>
  </consistency>
  
  <!-- === ТЕСТОВЫЕ СЦЕНАРИИ === -->
  <test-scenarios>
    
    <!-- Level 1: Детерминированные проверки -->
    <group level="deterministic">
      <scenario id="SC-001">
        <name>{название сценария}</name>
        <description>{описание}</description>
        <anchor ref="{ANCHOR_ID}"/>
        <given>{предусловия}</given>
        <when>{действие}</when>
        <then>
          <assertion type="equal">
            <actual>result.status</actual>
            <expected>200</expected>
          </assertion>
        </then>
      </scenario>
      
      <scenario id="SC-002">
        <!-- ... -->
      </scenario>
    </group>
    
    <!-- Level 2: Тесты траектории (log markers) -->
    <group level="trajectory">
      <scenario id="SC-010">
        <name>Проверка лог-маркеров для {ANCHOR_ID}</name>
        <anchor ref="{ANCHOR_ID}"/>
        <required-log-markers>
          <marker point="ENTRY">
            <pattern>[{MODULE}][{FUNCTION}][{ANCHOR_ID}][ENTRY]</pattern>
            <must-contain>
              <field>module</field>
              <field>function</field>
              <field>anchor</field>
            </must-contain>
          </marker>
          
          <marker point="EXIT">
            <pattern>[{MODULE}][{FUNCTION}][{ANCHOR_ID}][EXIT]</pattern>
            <must-contain>
              <field>result</field>
            </must-contain>
          </marker>
          
          <marker point="CHECK" optional="true">
            <pattern>[{MODULE}][{FUNCTION}][{ANCHOR_ID}][CHECK]</pattern>
          </marker>
          
          <marker point="DECISION" optional="true">
```

### Пример кода (AUTH_LOGIN)

```python
# [START_AUTH_LOGIN]
"""
ANCHOR: AUTH_LOGIN
PURPOSE: Аутентификация пользователя по email и паролю.

@PreConditions:
- email: валидный email формат
- password: непустая строка

@PostConditions:
- при успехе: { access_token, refresh_token, user }
- при ошибке: { error: "INVALID_CREDENTIALS" }

@Invariants:
- пароль никогда не возвращается в ответе
- access_token живёт 8 часов
- refresh_token живёт 30 дней

@SideEffects:
- создаёт/обновляет refresh_token в БД
- пишет лог попытки входа

@ForbiddenChanges:
- нельзя убрать rate limiting
- нельзя возвращать разные ошибки для "user not found" и "wrong password" (security)
"""
def login(email: str, password: str) -> dict:
    log_line("auth", "DEBUG", "login", "AUTH_LOGIN", "ENTRY", {
        "email": email,
        "has_password": bool(password),
    })
    
    # Rate limiting check
    if is_rate_limited(email):
        log_line("auth", "WARN", "login", "AUTH_LOGIN", "DECISION", {
            "decision": "rate_limited",
            "email": email,
        })
        log_line("auth", "DEBUG", "login", "AUTH_LOGIN", "EXIT", {
            "result": "rejected",
            "error": "RATE_LIMITED",
        })
        return {"error": "RATE_LIMITED"}
    
    # Find user
    user = User.objects.filter(email=email).first()
    log_line("auth", "DEBUG", "login", "AUTH_LOGIN", "CHECK", {
        "check": "user_exists",
        "result": user is not None,
        "email": email,
    })
    
    if not user:
        log_line("auth", "WARN", "login", "AUTH_LOGIN", "DECISION", {
            "decision": "invalid_credentials",
            "reason": "user_not_found",
        })
        log_line("auth", "DEBUG", "login", "AUTH_LOGIN", "EXIT", {
            "result": "rejected",
            "error": "INVALID_CREDENTIALS",
        })
        return {"error": "INVALID_CREDENTIALS"}
    
    # Validate password
    if not verify_password(password, user.password_hash):
        log_line("auth", "WARN", "login", "AUTH_LOGIN", "DECISION", {
            "decision": "invalid_credentials",
            "reason": "wrong_password",
        })
        log_line("auth", "DEBUG", "login", "AUTH_LOGIN", "EXIT", {
            "result": "rejected",
            "error": "INVALID_CREDENTIALS",
        })
        return {"error": "INVALID_CREDENTIALS"}
    
    # Generate tokens
    tokens = generate_tokens(user)
    
    log_line("auth", "INFO", "login", "AUTH_LOGIN", "STATE_CHANGE", {
        "entity": "session",
        "action": "created",
        "user_id": user.id,
    })
    
    log_line("auth", "DEBUG", "login", "AUTH_LOGIN", "EXIT", {
        "result": "success",
        "user_id": user.id,
    })
    
    return {
        "access_token": tokens["access"],
        "refresh_token": tokens["refresh"],
        "user": {"id": user.id, "email": user.email, "role": user.role},
    }
# [END_AUTH_LOGIN]
```
