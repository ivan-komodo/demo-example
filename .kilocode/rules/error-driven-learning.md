# Обучение на ошибках (Error-Driven Learning)

## Ключевой принцип

**Любая ошибка, замечание или неоднозначность — это сигнал для создания уточняющего правила. Система должна самообучаться, а не повторять одни и те же ошибки.**

---

## 1. Триггеры для создания правила

Правила создаются при:

| Триггер | Пример |
|---------|--------|
| **Ошибка в коде** | Нарушение контракта, баг, падение теста |
| **Замечание пользователя** | "Не то сделал", "Нужно по-другому" |
| **Нарушение стиля** | Не тот файл, не та структура |
| **Неоднозначность** | Агент ошибся в интерпретации |
| **Повторяющаяся проблема** | Одна ошибка >2 раз |
| **Code review замечание** | Отклонение на ревью |

---

## 2. Процесс создания правила

### 2.1 Алгоритм

```
ОШИБКА/ЗАМЕЧАНИЕ
       ↓
[Фиксация проблемы]
       ↓
[Анализ причины: почему произошло?]
       ↓
[Формулировка правила: что делать правильно?]
       ↓
[Классификация: к какой категории относится?]
       ↓
[Добавление в rules/*.md]
       ↓
[Обновление README.md]
       ↓
[Проверка: предотвращает ли правило повтор?]
```

### 2.2 Формат правила

```markdown
### RULE_[ID]: [Название]

**Проблема**: Что произошло (пример ошибки).

**Причина**: Почему агент ошибся.

**Решение**: Как нужно делать правильно.

**Пример**:
❌ Неправильно: ...
✅ Правильно: ...

**Применимость**: Когда применять это правило.

**Связанные RULE**: Другие правила по теме.
```

## 5. Обязанности ИИ-агента

### 5.1 При обнаружении ошибки

Агент ОБЯЗАН:

1. **Зафиксировать ошибку** — что именно не так
2. **Проанализировать причину** — почему принято неверное решение
3. **Сформулировать правило** — как избежать в будущем
4. **Добавить правило** в соответствующий файл
5. **Обновить README.md** — добавить файл/правило в список

### 5.2 Шаблон добавления правила

```markdown
### RULE_[CATEGORY]_[NUMBER]: [Название]

**Проблема**: [Дата: YYYY-MM-DD] [Описание ошибки]

**Причина**: [Почему агент ошибся — анализ]

**Решение**: 
1. [Шаг 1]
2. [Шаг 2]

**Пример**:
❌ Неправильно:
```python
# пример ошибки
```

✅ Правильно:
```python
# как должно быть
```

**Применимость**: [Когда применять]

**Дата добавления**: YYYY-MM-DD
**Инцидент**: [Ссылка на коммит/PR/чат]
```

## 6. Примеры правил

### Пример 1: Нарушение контракта

```markdown
### RULE_CONTRACT_001: Проверять Invariants перед изменениями

**Проблема**: [2024-01-15] Агент добавил новый побочный эффект, не указав его в контракте.

**Причина**: Агент не проверил @SideEffects перед изменением кода.

**Решение**: 
1. Перед любым изменением процедуры/функции — прочитать контракт
2. Проверить @SideEffects и @Invariants
3. Если новое поведение не подходит — обновить контракт ИЛИ отказаться от изменения

**Пример**:
❌ Неправильно:
```python
def process_order(order_id):
    send_email(user_id)  # новый побочный эффект без контракта
    return result
```

✅ Правильно:
```python
# Обновить контракт: @SideEffects: отправка email пользователю
def process_order(order_id):
    send_email(user_id)
    return result
```

**Применимость**: При любом изменении существующей функции с контрактом.
```

### Пример 2: Неправильный файл

```markdown
### RULE_ARCH_001: Функции размещения в apps/

**Проблема**: [2024-01-16] Агент создал функцию в core/utils.py вместо apps/users/services.py.

**Причина**: Агент не проверил структуру проекта в project-context.md.

**Решение**:
1. Перед созданием нового файла — проверить структуру в project-context.md
2. Бизнес-логика -> apps/[domain]/services.py
3. Общие утилиты -> core/utils.py только если не привязаны к домену

**Пример**:
❌ Неправильно:
- core/utils.py: def calculate_user_discount()

✅ Правильно:
- apps/users/services.py: def calculate_user_discount()
```

## 7. Интеграция с существующими правилами

### 7.1 Связка с контрактами

Каждое правило из `error-patterns.md` должно ссылаться на соответствующий раздел:

```markdown
**Связанные разделы**:
- semantic-code-markup.md#5-запреты-на-разрушительный-рефакторинг
- ai-logging.md#3-шаблон-логирования-в-функции
```

### 7.2 Обновление чеклистов

При добавлении нового правила — обновить чеклист в `semantic-code-markup.md`:

```markdown
## 10. Чеклист перед сдачей кода

- [ ] Проверено RULE_CONTRACT_001 (Инварианты)
- [ ] Проверено RULE_LOG_001 (EXIT логи)
- [ ] ... (добавить новые правила)
```

## 8. Метрики качества

### 8.1 Целевые показатели

| Метрика | Цель |
|---------|------|
| Повторение одной ошибки | ≤ 1 раз |
| Правил на категорию | 5-20 (поддерживать актуальность) |
| Устаревшие правила | Удалять/актуализировать |

### 8.2 Ревизия правил

Раз в месяц:
1. Проверить актуальность правил
2. Удалить дубликаты
3. Объединить похожие
4. Актуализировать примеры

## 9. Ответственность

### 9.1 Кто добавляет правила

- **ИИ-агент** — автоматически при обнаружении ошибки
- **Пользователь** — при замечаниях и code review
- **Code Reviewer** — при отклонении PR

### 9.2 Формат сообщения при добавлении

```
📝 Правило добавлено: RULE_[CATEGORY]_[NUMBER]
   Файл: .kilocode/rules/error-patterns.md
   Причина: [краткое описание ошибки]
```
