# Семантическая разметка (обязательные правила)

## Задача

Сделать так, чтобы в каждом месте было **однозначно**: где начинается и **где заканчивается** смысловой блок, и **какой у него контракт**. Локализация без парных границ и без `@PreConditions` / `@PostConditions` считается **ошибкой разметки** (её чаще всего выдают LLM по привычке).

## Один идентификатор на блок

Для **каждой** функции, **каждого** метода класса, **каждого** метода объекта с именем:

1. Выбери **`ANCHOR_ID`** — `UPPER_SNAKE_CASE`, уникальный в пределах файла (лучше — по смыслу: `ATTACH_TAB`, не `NETWORK_REGISTRY`).
2. **`ANCHOR_ID` должен совпадать буквально** во всех местах:
   - строка `// [START_ANCHOR_ID]` (или эквивалент в другом языке);
   - строка `ANCHOR: ANCHOR_ID` внутри блока комментария контракта;
   - строка `// [END_ANCHOR_ID]` после закрывающей `}` функции;
   - аргумент **`anchor`** в `logLine(...)` для этого тела функции (см. `.kilocode/rules/ai-logging.md`).

**Запрещено:** вести логи под якорём «модуля» (`CDP_NETWORK_CAPTURE`, `OPENAI_COMPAT`), а в комментарии функции указать другой `ANCHOR` (`ATTACH_TAB`, `RUN_CHAT_COMPLETION`). Это ломает связку «контракт ↔ лог».

## Обязательный каркас вокруг тела (TypeScript / JavaScript)

Используй **тройку**: линейный комментарий с `[START_…]`, многострочный **контракт**, тело функции, линейный `[END_…]`.

Шаблон (копируй и подставляй `ANCHOR_ID`):

```typescript
// [START_ANCHOR_ID]
/*
 * ANCHOR: ANCHOR_ID
 * @PreConditions:
 * - (явные входные условия; если тривиально — строка «нет нетривиальных предусловий»)
 * @PostConditions:
 * - (что гарантируется при успехе; границы ошибок — при необходимости отдельным пунктом)
 * PURPOSE: Одна фраза: зачем существует эта функция.
 */
async function example(): Promise<void> {
  // тело
}
// [END_ANCHOR_ID]
```

Требования:

- **`[START_ANCHOR_ID]`** и **`[END_ANCHOR_ID]`** — **обязательны** для каждой именованной функции/метода. Один `ANCHOR_ID` — одна пара скобок.
- Внутри контракта **обязательны** строки `ANCHOR:`, `@PreConditions:`, `@PostConditions:`, `PURPOSE:` (заголовки секций не опускай; содержимое может быть коротким).
- Опционально: `@see: Module -> symbol()` — если без этого связь с кодовой базой неочевидна.

## Явно запрещённые «упрощения» (типичный сбой LLM)

Не считай разметку выполненной, если сделано хоть одно из нижеследующего:

- Только блок вида `/* ANCHOR: … PURPOSE: … */` **без** пары `// [START_…]` / `// [END_…]` и **без** `@PreConditions` / `@PostConditions`.
- `END_ANCHOR` как текст комментария **вместо** пары `[END_ANCHOR_ID]` с **тем же** id, что и `[START_…]`.
- Контракт висит над **классом**, но **у методов** нет своих пар `[START_…]`…`[END_…]` (каждый метод — отдельный контракт).
- Разные строки-идентификаторы в START, в `ANCHOR:` и в END (опхечатки, «общий» якорь файла vs якорь функции).

## Файл целиком (опционально)

Если файл длинный или несколько подсистем в одном месте — добавь **чанк** (см. примеры в репозитории), но **чанк не заменяет** контракт на каждую функцию.

## Чеклист перед сдачей патча

- У **каждой** новой/изменённой функции или метода есть `// [START_…]` … `// [END_…]` и полный контракт.
- `ANCHOR_ID` в START, в `ANCHOR:` и в END **совпадает**.
- Тот же `ANCHOR_ID` передаётся в `logLine` для этой функции (см. правило логирования).
