# Семантический граф (XML)

## Файл-источник правды

- Один канонический файл: **`.kilocode/semantic-graph.xml`**.
- Любое существенное изменение архитектуры, новый модуль, новая связь messaging/API — **обновляй этот файл в том же PR/коммите**, что и код.

## Зачем XML

- Иерархия = контекст; **парные теги** задают границы смысловых блоков.
- **`id`** на узле графа — стабильный якорь для ссылок (`ref`, `from`/`to`); не переименовывай `id` без нужды; при переименовании обнови все ссылки.
- Атрибуты — факты (пути, типы); текст внутри тега — краткое **человеческое** описание намерения.

## Обязательные разделы (поддерживай в актуальном виде)

1. **`<meta>`** — продукт, назначение, ограничения окружения (Manifest V3 и т.д.).
2. **`<components>`** — каждый логический блок: `id`, `kind`, при необходимости `path`; вложенные `<role>`, `<depends-on ref="..."/>`, `<exposes>` / `<consumes>`.
3. **`<relationships>`** — рёбра: **`<edge id="..." from="..." to="..." kind="..."/>`** (сообщения, импорты, IPC, события DOM — `kind` из согласованного словаря).
4. **`<data-flow>`** (опционально) — потоки данных между компонентами.
5. **`<decisions>`** — зафиксированные архитектурные решения (ADR в miniature): `id`, `status`, краткий `rationale`.

## Правила качества

- **`id` уникален** в документе; для вложенности используй префиксы: `ext.background`, `page.options`.
- Не дублируй длинные описания кода — в XML **смысл и связи**, детали реализации остаются в исходниках.
- После крупного рефакторинга: пройдись по `depends-on` / `edge` и удали мёртвые ссылки.

## Мини-чеклист перед merge

- [ ] Добавлены/обновлены узлы для новых частей.
- [ ] Обновлены рёбра и зависимости.
- [ ] `meta` и при необходимости `decisions` отражают текущее состояние.

При первом появлении кодовой базы заполни `.kilocode/semantic-graph.xml` по шаблону из файла; не оставляй плейсхолдеры без пояснения в `<meta><note>`.
